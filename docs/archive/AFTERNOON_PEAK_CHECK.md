# 14:00 Afternoon Peak-Window Boost Feature

**Status**: ✅ Complete and Ready to Deploy

## Overview

The 14:00 Afternoon Peak-Window Check is a new automated feature that runs daily at 14:00 to decide if the battery should be boosted to survive the 16:00-19:00 peak pricing window. This feature complements the existing:
- **22:00 Overnight Charging** (growatt_charger.py): Prepares battery for overnight consumption
- **05:00 Morning SOC Check** (morning_soc_check.py): Captures actual overnight SOC drop
- **14:00 Afternoon Peak Check** (afternoon_peak_check.py): NEW - Boost for peak window protection

## Architecture

### Entry Points
- **`afternoon_peak_check.py`** (root): Simple wrapper script
  - Imports and calls `src.app_afternoon_peak_check.main()`
  - Called by Task Scheduler at 14:00 via `run_afternoon_peak_check.bat`

### Business Logic
- **`src/app_afternoon_peak_check.py`**: Orchestration logic
  - `AfternoonPeakChecker` class: Checks forecast, current SOC, determines boost need
  - Imports `modules.peak_window_boost` for decision logic
  - Applies settings via `GrowattAPI.update_charge_settings()`
  - Logs decisions to `output/peak_decisions.csv` for analysis

### Shared Utilities
- **`modules/peak_window_boost.py`**: Decision logic
  - `should_boost_battery_for_peak_window()`: Returns (bool, reason, details)
  - `calculate_peak_window_boost_target()`: Returns target SOC if boost needed
  - Based on: remaining forecast, current SOC, battery capacity, average load

### Task Scheduler Integration
- **`create_afternoon_peak_check_task.ps1`**: Registers scheduled task
  - Requires: Run as Administrator
  - Creates daily task at 14:00
  - Calls: `run_afternoon_peak_check.bat`

- **`run_afternoon_peak_check.bat`**: Task execution wrapper
  - Activates virtual environment
  - Calls: `python afternoon_peak_check.py conf/growatt-charger.ini`

### Analysis & Review
- **`review_peak_decisions.py`**: Analyzes boost effectiveness
  - Summary statistics: total boosts, not-boosted decisions
  - Forecast accuracy analysis
  - Boost effectiveness (did it prevent undersupply?)
  - Recommendations for threshold tuning
  - Exports: `output/peak_decisions_analysis.csv`

## Decision Logic

### Input Parameters
At 14:00, the system checks:
1. **Remaining Forecast** (Wh): Solar generation from now until sunset (~4-6 hours)
2. **Current Battery SOC** (%): Present charge level
3. **Battery Capacity** (Wh): From config
4. **Average Load** (W): From config (default 850W)

### Calculation
```
Peak Window Duration = 3 hours (16:00-19:00)
Peak Consumption = Average Load × Peak Window Duration
Peak Generation = Remaining Forecast × (Peak Window hours / Remaining hours)
Peak Shortfall = Peak Consumption - Peak Generation

Required SOC (%) = Battery % needed to cover shortfall
```

### Decision Rules
**BOOST if**:
- Peak Shortfall > Battery Headroom
- OR Required SOC > 10% margin above current SOC

**NO BOOST if**:
- Battery sufficient to cover peak window
- AND Forecast adequate for evening/overnight consumption

### Boost Application
If boost needed:
- Set charge rate: 80% (aggressive but sustainable)
- Target SOC: Calculated to cover peak window + headroom
- Duration: 14:00-16:00 (2 hours to prepare for peak)
- Override any existing 22:00 schedule

## Installation & Configuration

### 1. Prerequisites
- Python 3.8+ with modules: requests, configparser, growattServer
- Windows Task Scheduler (built-in)
- Virtual environment: `.venv/` (existing setup)

### 2. Register Scheduled Task
```powershell
# Run as Administrator
.\create_afternoon_peak_check_task.ps1
```

This creates:
- Task Name: `GrowattAfternoonPeakCheck`
- Schedule: Daily at 14:00
- User: Current Windows user
- Action: Execute `run_afternoon_peak_check.bat`

### 3. Verify Installation
```powershell
Get-ScheduledTask -TaskName "GrowattAfternoonPeakCheck"
```

Should show:
```
TaskName                       State   Enabled
--------                       -----   -------
GrowattAfternoonPeakCheck      Ready   True
```

### 4. Manual Testing
```powershell
# Test the script directly
python afternoon_peak_check.py conf\growatt-charger.ini
```

Expected output:
```
2024-11-15 14:00:00 INFO 14:00 Peak-Window Check: Forecast 5.0kWh, SOC 35%, Decision: BOOST
```

## Configuration

### Config File: `conf/growatt-charger.ini`

No new config required. Uses existing sections:

**[growatt]**
- `battery_capacity_wh`: Used to calculate shortfall percentage
- `average_load_w`: Used to calculate peak consumption (default: 850W)
- `maximum_charge_pct`: Limits boost target SOC

**[forecast]**
- `primary_provider`: Which forecast to use (Solcast, Forecast.Solar, etc.)

**[growatt] or [tariff] (future)**
- Could add `peak_window_start` (16:00) and `peak_window_hours` (3) for customization
- Currently hardcoded in `modules/peak_window_boost.py`

### Troubleshooting

**Task doesn't run at 14:00**
- Check Task Scheduler: `taskschd.msc`
- Verify task is enabled and user has permission
- Check event log: Event Viewer → Windows Logs → System

**API authentication failures**
- Verify Growatt credentials in config or environment variables
- Check API connectivity: `python -c "from growattServer.api import APIClient; print(APIClient())"`

**No peak_decisions.csv file**
- Script may have crashed; check logs/afternoon-peak-check.log
- Verify output/ directory exists and is writable

## Performance & Impact

### Execution Time
- ~5-10 seconds per run (mostly API calls)
- Negligible CPU/memory impact

### API Calls
- 1x Login (Growatt API)
- 1x Get System Status (fetch current SOC)
- 1x Get Forecast (forecast provider)
- 1x Update Charge Settings (if boost needed) — optional

### Battery Impact
- Only boosts when necessary (peaks forecasts < 8kWh average)
- 2-hour boost window (14:00-16:00) is gentle on battery cycles
- Reduces grid demand during peak pricing window

## Limitations & Future Work

### Current Limitations
1. **Hardcoded Peak Window**: 16:00-19:00 always; should read from config
2. **Simplified Remaining Forecast**: Uses full-day forecast; could use hourly API
3. **Static Load Assumption**: Average load (850W) is constant; could be dynamic
4. **No Tariff Integration**: Peak window and rates are assumed; should read from tariff API

### Planned Enhancements
1. **Configurable Peak Window**: Move 16:00-19:00 to config file
2. **Hourly Forecasts**: Use provider's hourly data for accurate remaining calculation
3. **Dynamic Load Profiles**: Read expected load from home automation data
4. **Tariff API Integration**: Fetch peak pricing window from utility provider
5. **Multi-Window Support**: Handle multiple peak windows (morning + evening)
6. **ML-Based Tuning**: Learn optimal thresholds from historical effectiveness data
